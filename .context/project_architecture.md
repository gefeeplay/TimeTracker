# Архитектура проекта

## Проект: Трекер экранного времени (Windows)

---

# 1. Общая архитектурная модель

Все компоненты реализованы в рамках одного проекта с логическим разделением по папкам.

## Структура проекта

```
TimeTracker
 ├── Models
 ├── Services
 ├── Data
 ├── Monitoring
 ├── ViewModels
 └── Views
```

Основная идея — минимизация избыточных абстракций при сохранении читаемости и разделения ответственности.

---

# 2. Models

## Назначение

Папка Models содержит модели данных приложения.

Это простые классы, отражающие структуру базы данных и используемые во всех частях системы.

## Содержимое

- Application
- Category
- UsageSession
- DailyAppStat

## Ответственность

- Представление структуры данных
- Хранение свойств сущностей
- Минимальная логика (при необходимости)

Модели не содержат сложной бизнес-логики.

---

# 3. Services

## Назначение

Services — это основной слой бизнес-логики приложения.

## Содержимое

- UsageService
- StatisticsService
- CleanupService

## Ответственность

- Обработка данных мониторинга
- Работа с базой данных
- Расчёт статистики
- Агрегация данных
- Очистка устаревших записей

Сервисы напрямую взаимодействуют с Data-слоем.

---

# 4. Data

## Назначение

Data отвечает за доступ к базе данных SQLite.

## Содержимое

- DbContext
- Конфигурация подключения
- SQL-запросы / ORM-настройки

## Ответственность

- Сохранение данных
- Чтение данных
- Выполнение запросов

В данной архитектуре Data не выделен в отдельный слой с интерфейсами — используется напрямую сервисами.

---

# 5. Monitoring

## Назначение

Monitoring отвечает за сбор информации об активности пользователя.

Это системный компонент, взаимодействующий с операционной системой.

## Содержимое

- ActivityTracker
- Работа с Win32 API

## Ответственность

- Определение активного окна
- Отслеживание переключения приложений
- Фиксация времени активности
- Формирование данных для записи

## Взаимодействие

Monitoring передаёт данные напрямую в UsageService.

Схема:

```
Monitoring → UsageService → SQLite
```

---

# 6. ViewModels

## Назначение

ViewModels реализуют паттерн MVVM и выступают связующим звеном между UI и логикой приложения.

## Содержимое

- MainViewModel
- CategoriesViewModel
- DashBoardViewModel
- SettingsViewModel

## Ответственность

- Подготовка данных для отображения
- Вызов сервисов
- Обработка команд UI
- Управление состоянием интерфейса

ViewModels не содержат логики работы с базой данных напрямую.

---

# 7. Views

## Назначение

Views — пользовательский интерфейс приложения.

## Содержимое

- XAML-страницы
- Окна приложения

## Ответственность

- Отображение данных
- Взаимодействие с пользователем

Views работают через привязку данных (Binding) с ViewModels.

---

# 8. Взаимодействие компонентов

## Сбор данных

```
Monitoring
   ↓
UsageService
   ↓
Data (SQLite)
```

## Отображение данных

```
View
   ↓
ViewModel
   ↓
StatisticsService
   ↓
Data (SQLite)
```
