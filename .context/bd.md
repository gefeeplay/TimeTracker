# Структура базы данных

## Проект: Трекер экранного времени (Windows, SQLite)

---

# 1. Общая концепция

База данных предназначена для локального хранения информации об использовании приложений пользователем.

Требования:

- Локальное хранение (SQLite)
- Минимальное потребление ресурсов
- Поддержка хранения данных 6–12 месяцев
- Возможность агрегации статистики
- Расширяемость архитектуры

Используемые пакеты:

- Microsoft.Data.Sqlite
- Dapper

Архитектура хранения данных разделена на:

1. Справочные таблицы
2. Таблицы сырых событий (сессии)
3. Таблицы агрегированной статистики

---

# 2. Диаграмма сущностей (логическая модель)

Categories (1) ────< Applications (1) ────< UsageSessions
│
└────< DailyAppStats

---

# 3. Таблицы базы данных

---

## 3.1 Categories

Справочник категорий приложений.

### Назначение

Позволяет группировать приложения (Работа / Развлечение).

### Структура

| Поле | Тип     | Обязательное | Описание                       |
| ---- | ------- | ------------ | ------------------------------ |
| Id   | INTEGER | Да           | Первичный ключ (AUTOINCREMENT) |
| Name | TEXT    | Да           | Название категории (UNIQUE)    |

### Ограничения

- UNIQUE(Name)

---

## 3.2 Applications

Справочник отслеживаемых приложений.

### Назначение

Хранит информацию о каждом уникальном процессе.

### Структура

| Поле        | Тип      | Обязательное | Описание                                     |
| ----------- | -------- | ------------ | -------------------------------------------- |
| Id          | INTEGER  | Да           | Первичный ключ                               |
| ProcessName | TEXT     | Да           | Имя процесса (например, chrome.exe)          |
| DisplayName | TEXT     | Нет          | Отображаемое имя (Google Chrome)             |
| CategoryId  | INTEGER  | Да           | Внешний ключ на Categories                   |
| IsActive    | INTEGER  | Да           | Флаг активности (1 — активно, 0 — исключено) |
| CreatedAt   | DATETIME | Да           | Дата добавления                              |

### Связи

- CategoryId → Categories(Id)

### Индексы

- INDEX(ProcessName)
- INDEX(CategoryId)

---

## 3.3 UsageSessions

Основная таблица хранения сырых данных активности.

### Назначение

Фиксирует каждую сессию использования приложения.

### Структура

| Поле            | Тип      | Обязательное | Описание                     |
| --------------- | -------- | ------------ | ---------------------------- |
| Id              | INTEGER  | Да           | Первичный ключ               |
| ApplicationId   | INTEGER  | Да           | Внешний ключ на Applications |
| StartTime       | DATETIME | Да           | Время начала сессии          |
| EndTime         | DATETIME | Да           | Время окончания              |
| DurationSeconds | INTEGER  | Да           | Длительность в секундах      |
| CreatedAt       | DATETIME | Да           | Время записи в БД            |

### Связи

- ApplicationId → Applications(Id)

### Индексы

- INDEX(ApplicationId)
- INDEX(StartTime)
- INDEX(EndTime)

### Примечание

Таблица может очищаться автоматически (хранение не более 6 месяцев).

---

## 3.4 DailyAppStats

Таблица агрегированной статистики по дням.

### Назначение

Ускоряет построение аналитики и отчетов.

### Структура

| Поле                 | Тип      | Обязательное | Описание                   |
| -------------------- | -------- | ------------ | -------------------------- |
| Id                   | INTEGER  | Да           | Первичный ключ             |
| ApplicationId        | INTEGER  | Да           | Внешний ключ               |
| Date                 | DATE     | Да           | Дата (без времени)         |
| TotalDurationSeconds | INTEGER  | Да           | Суммарное время за день    |
| SessionsCount        | INTEGER  | Да           | Количество сессий за день  |
| UpdatedAt            | DATETIME | Да           | Время последнего пересчета |

### Связи

- ApplicationId → Applications(Id)

### Ограничения

- UNIQUE(ApplicationId, Date)

### Индексы

- INDEX(Date)
- INDEX(ApplicationId)

---

# 4. Стратегия хранения данных

Рекомендуемая политика:

- UsageSessions — хранить 6 месяцев
- DailyAppStats — хранить 12 месяцев

Очистка выполняется фоновым сервисом.

Пример логики очистки:

- Удаление UsageSessions старше 6 месяцев
- Удаление DailyAppStats старше 12 месяцев

---

# 5. Принципы проектирования

1. Нормализация справочников
2. Хранение сырых данных для гибкости аналитики
3. Использование агрегированных таблиц для производительности
4. Минимизация JOIN в UI-запросах
5. Индексация по датам и внешним ключам
